## 課題１

### シングルテナントモデルを採用する

[ASIL](https://zenn.dev/praha/articles/asil_programming)で考える。

**曝露可能性(Exposure)** 

開発現場が大忙しでレビューを通さずにデプロイされる可能性がある＋ドメインへの理解が浅い新人が開発に加わる、など今回の失態が発生する可能性は十二分にあると考えられる。


**重大度(severity)**

間違った実装をしてしまうと多額の賠償金を請求され、事業停止に追い込まれるレベルなので重大度は高いと言える。

**制御可能性(controllability)**

すべての会社の従業員データが漏れてしまったら、秘密保持などの契約違反で一発アウトだと思うので制御可能性は低いと言える。

上記を踏まえると、テナントの分離度合いの高い[シングルテナントモデル](https://learn.microsoft.com/ja-jp/azure/architecture/isv/application-tenancy#single-tenant)
を採用するのが良いと考える。


## 課題2

### 「テナント毎にデータベースを分割する」パターン

**メリット**

**デメリット**

### 「テナント毎にスキーマを分割する」パターン

**メリット**

- テナントの分離度合いが高い
- 他のテナントからの影響を受けない
- スキーマをテナントごとにカスタマイズや最適化ができる

**デメリット**

### 「すべてのテナントで同じスキーマを使う」パターン

**メリット**

- スケールしやすい（アプリケーションでサービスを提供できる同時顧客 (またはテナント) の数を増やしやすい）
- テナントあたりのコストが低い

**デメリット**

- テナントの分離度合いが低い
- あるテナントが過稼働状態だと別のテナントのパフォーマンスに影響を及ぼしうる
